# Contracts

The Shade Protocol smart contracts run on Citrea mainnet (chain ID 4114). They manage the privacy pool, Merkle tree, proof verification, and token handling.

**Source:** [github.com/shadeprotocolcom/contracts](https://github.com/shadeprotocolcom/contracts)

## Contract architecture

```
ShadePool.sol          — Entry point: shield() + transact()
  ├── Commitments.sol  — Binary Poseidon Merkle tree (depth 16)
  ├── Verifier.sol     — Generated by snarkjs (Groth16 verifier)
  ├── TokenGuard.sol   — cBTC/WcBTC handling: auto-wrap / auto-unwrap
  ├── PoseidonT3.sol   — 2-input Poseidon hash (from zk-kit, MIT)
  └── PoseidonT4.sol   — 3-input Poseidon hash (from zk-kit, MIT)

ShadeKeyRegistry.sol   — On-chain public key registry (independent)
```

## ShadePool

The main entry point. Provides two public functions:

### `shield(ShieldRequest[] calldata requests)`

Deposits native cBTC or pre-approved WcBTC into the private pool.

- Accepts native cBTC as `msg.value` — the contract auto-wraps to WcBTC via `TokenGuard`
- Computes the commitment hash: `PoseidonT4(npk, tokenHash, value)`
- Inserts commitment(s) into the Merkle tree
- Emits `Shield` event with commitment preimages and encrypted ciphertext

### `transact(Transaction[] calldata transactions)`

Executes one or more private transactions. Each transaction:

1. Validates the chain ID matches `4114` (prevents cross-chain replay)
2. Verifies the claimed Merkle root exists in `rootHistory`
3. Checks and marks nullifiers as spent (reverts on double-spend)
4. Builds 6 public inputs: `[merkleRoot, boundParamsHash, null0, null1, comm0, comm1]`
5. Verifies the Groth16 proof via the `Verifier` contract
6. Inserts new commitments into the Merkle tree
7. Emits `Transact` and `Nullified` events
8. If `unshield` flag is set: verifies the unshield preimage and sends native cBTC to the recipient

## Commitments

Manages the binary Poseidon Merkle tree:

- **Depth**: 16 (capacity: 65,536 leaves)
- **Hash function**: Poseidon (via PoseidonT3)
- **Zero value**: `keccak256("Shade") % SNARK_SCALAR_FIELD`
- **Root history**: stores all historical roots so proofs remain valid even after new insertions
- **Nullifier tracking**: `mapping(uint16 => mapping(bytes32 => bool))` per tree number

## TokenGuard

Handles the conversion between native cBTC and WcBTC (WETH9-style wrapped token):

- `_handleShieldDeposit()`: wraps native cBTC to WcBTC, or pulls pre-approved WcBTC
- `_handleUnshield()`: unwraps WcBTC back to native cBTC and sends to the recipient

Users never interact with WcBTC directly — the wrapping is transparent.

## Verifier

Generated by `snarkjs groth16 exportsolidityverifier`. Contains:

- BN254 pairing operations
- The verifying key (embedded as constants)
- `verifyProof(uint[2] a, uint[2][2] b, uint[2] c, uint[6] input) → bool`

The verifier expects exactly 6 public inputs matching the circuit's public signal layout.

## Events

| Event | Emitted by | Data |
|---|---|---|
| `Shield` | `shield()` | treeNumber, startPosition, commitment preimages, ciphertexts, fees |
| `Transact` | `transact()` | treeNumber, startPosition, commitment hashes, ciphertexts |
| `Nullified` | `transact()` | treeNumber, nullifier hashes |

These events are indexed by the [Indexer](/infrastructure/indexer) for wallet balance scanning.

## Data types

```solidity
struct CommitmentPreimage {
    bytes32 npk;
    TokenData token;
    uint120 value;
}

struct TokenData {
    uint8 tokenType;      // 0 = ERC20
    address tokenAddress;  // WcBTC address
    uint256 tokenSubID;   // 0 for ERC-20
}

struct Transaction {
    SnarkProof proof;
    bytes32 merkleRoot;
    bytes32[] nullifiers;
    bytes32[] commitments;
    BoundParams boundParams;
    CommitmentPreimage unshieldPreimage;
}

struct BoundParams {
    uint16 treeNumber;
    UnshieldType unshield;  // NONE = 0, NORMAL = 1
    uint64 chainID;
    CommitmentCiphertext[] commitmentCiphertext;
}
```

## ShadeKeyRegistry

On-chain registry for Shade public keys. Enables fully non-custodial operation by removing the dependency on the centralized indexer for recipient key lookups.

### `registerKeys(bytes32 viewingPubKeyX, bytes32 viewingPubKeyY, bytes32 masterPublicKey)`

Register or update your Shade public keys. Authenticated via `msg.sender` — only the account owner can set their own keys.

- Overwrites are allowed (e.g. after wallet recovery)
- Reverts if `masterPublicKey` is zero
- Emits `KeysRegistered(address indexed account, bytes32, bytes32, bytes32)`

### `getKeys(address account) → (bytes32, bytes32, bytes32)`

Look up a user's registered public keys. Free `view` function — no gas cost.

### `isRegistered(address account) → bool`

Check whether an address has registered keys. Returns `true` if `masterPublicKey` is non-zero.

### Gas costs

| Operation | Gas | Cost at 1 gwei |
|---|---|---|
| First registration (3 cold SSTOREs) | ~80K | ~0.00008 cBTC |
| Key update (3 warm SSTOREs) | ~15K | ~0.000015 cBTC |
| Lookup / isRegistered (eth_call) | 0 | Free |

cd packages/contracts

# Build
forge build

# Run tests
forge test -vvv

# Deploy (requires private key in environment)
forge script script/Deploy.s.sol --rpc-url https://rpc.citreascan.com --broadcast
